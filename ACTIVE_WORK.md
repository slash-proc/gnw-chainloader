## Debugging Log
- **i18n OCR non-Latin matching (#15) is not a font-metric problem (the obvious lead is a red herring).** *(Deprioritized 2026-06-03: superseded by forthcoming plans; #11 and #15 on hold.)* The OCR (`scripts/common/ocr.py`) loads the cooked `.fnt` glyphs and template-matches the screen; non-Latin (CJK / Greek / Cyrillic) does not match. The cooked CJK fonts differ from `latin.fnt` in cell metrics (`latin.fnt` cellH=12, ref_top=0; ja/zh/ko cellH=11, ref_top=1), which looked like the cause. But both the OCR (`ocr.py:86`) and the device font renderer (`src/modules/language/font_ext.c:71`) compute the per-glyph draw offset with the **same** formula `yoff = ref_top - GUI_FONT_REF_TOP` (`GUI_FONT_REF_TOP=1`), so the device and the template align identically. The metric difference is handled consistently on both sides and is NOT the cause. Greek and Cyrillic live in `latin.fnt` (same metrics as ASCII) yet also fail, confirming the cause is not metric-related. Diagnosis needs a device experiment: drive the device into a non-Latin language (chicken-and-egg, since the broken OCR cannot verify the language; cycle a known step count from English in code-sorted order, or read the active code another way), screenshot, then `locate()` the expected string and inspect the match score and which glyph the greedy `_best_glyph` matcher picks (Cyrillic 'Р' / Greek 'Α' look like Latin 'P' / 'A', a likely confusion). #11 (CJK launch-verb OCR) is blocked on this. Note: the language detector was repointed off the moved `g_current`/`g_langs` symbols to endonym-based OCR in `d9558c1`, so the Latin-language OCR tests work; only non-Latin remains.
- **ABI gating + installer-as-a-module — gates + transient installer module landed + verified.** Bumping `STRINGS_ABI_VERSION` (a boot-selector string change) and flashing only the core left the device running an ABI-3 core against the **stale ABI-2 language module + packs** on LittleFS → garbled non-English text. Principle: ABI is one compatibility contract gating **both load and install**, for **both** PIE modules (`MODULE_ABI_VERSION`) and `.lang` packs (`STRINGS_ABI_VERSION`); reject any mismatch in either direction and fall back to safe defaults; gating lives **in-core** (recovery-critical, must work with no module present). **Landed + hardware-verified (`81954a0`, `33fb308`):** in-core load gates (`loader.c` `mod_load_image`, `vfs_read_module`); the strings-ABI gate made **core-authoritative** (`vfs.c` enforces the core's own `STRINGS_ABI_VERSION`, not the `want_abi` a module passes — a stale module + stale packs used to "agree" and load); `module_header_t.abi` + a load-model `flags` field (resident vs `MOD_FLAG_TRANSIENT`); `mod_pool_mark`/`mod_pool_reset` (safe slot-reclaim, replacing the unsafe blanket `mod_reset`); and a shared `src/common/abi.h` (`module_abi_ok`/`pack_abi_ok`) the firmware **and** a host unit test (`scripts/build/test_abi_gate.c`, `make test_host`) both call — rejects abi±1 + bad magic for both header types (8/8). Verified on device: stale packs → clean English fallback (no garbage), then re-push restored languages.
