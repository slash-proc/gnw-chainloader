# Changelog

A terse, bullet-point record of completed bug fixes and tasks, newest first. Keep each entry to one line and link the commit(s) that landed it. (Entries below predate this convention, so they are not all commit-linked.)

- **Retro-Go launcher boot-time CRC gate (partial/corrupt-flash protection).** Mirrors the OFW CRC gate: `scripts/build/gen_retrogo_crc.py` (`make retrogo-crc`, after `make build-rg-sd`) bakes the CRC-32/MPEG-2 of the Retro-Go internal-flash image (the bytes flashed at `RETROGO_BASE`) into committed `src/common/retrogo_crc.h`; the chainloader re-checks it on-device with the hardware CRC unit (`retrogo_crc_ok`/`retrogo_bootable` in `system/ofw_verify.c`) before launching Retro-Go. A mismatch (interrupted/partial flash) falls back to the chainloader menu instead of jumping into an unbootable launcher (STABILITY IS LAW). Applied at the B-button, standby, boot-target, fast-boot, and menu launch paths; the inviolable CORE/RESET "Return to Main Menu" re-launch is intentionally NOT gated, and the check fails open when the header is unbaked (`RETROGO_CRC_LEN==0`) so a tree without a regenerated header is never bricked.
- **Retro-Go SD ROM cache is boxed into its own region; it no longer corrupts the OFW backups, FAT store, or LittleFS.** In SD-card mode, launching a game copies the ROM from SD into raw external flash via the round-robin allocator (`retro-go-sd/Core/Src/gw_flash_alloc.c`), which reserved only `get_ofw_extflash_size()` bytes at the bottom (the single active OFW's footprint, from its vector-table metadata) and wrote ROMs upward across the rest of the chip — straight over the Zelda/Mario OFW backups (`0x500000`/`0x520000`), the FAT module store (`0x540000`), and the LittleFS, none of which Retro-Go knows about (the recurring Mario-backup corruption, and a silent LittleFS hazard). Fix: the allocator now honors the larger of `get_ofw_extflash_size()` and the chainloader's `EXTFLASH_OFFSET` (exposed to C as `__EXTFLASH_OFFSET__` in `gw_linker.h`) as its lower bound, for both the write base and the capacity bound (cache metadata self-invalidates on the changed base); the upstreamable Retro-Go change is just "respect `EXTFLASH_OFFSET`". External flash is resliced so the LittleFS is a fixed 10 MiB partition butted against the FAT store (`MODULE_LFS_SIZE` at `MODULE_LFS_OFFSET_SD` = `0xA00000`..`0x1400000`) and the ROM cache occupies everything above it (`RETROGO_CACHE_OFFSET` = `0x1400000`..end), passed to the Retro-Go build as `EXTFLASH_OFFSET` so it can never reach the reserved regions. The Partition Viewer registers the bounded LittleFS and a labeled `Retro-Go Cache` region at their fixed spans and skips the cache during the sweep (raw ROM bytes can false-match FS magics). Host LFS tools (`push_batched.py`, `envprobe.py`, `provision.py`) pass the gnwmanager filesystem offset for the relocated LittleFS via the shared `lfs_gnwmanager_offset()` helper. Also hardened `STM32H7B0VBTx_SDCARD.ld` to apply `__EXTFLASH_OFFSET__` to `__EXTFLASH_START__` (matching the non-SD script).

- **Battery level detection fixed: reads the real charge again and matches Retro-Go/OFW (hardware-verified).** The hand-rolled register ADC init in `board_adc_init` (`board.c`) had drifted from the HAL `ADC_Init`/`ADC_ConfigChannel` path it replaced, accumulating four defects. (1) **Stuck at 0%:** it selected the ADC's *synchronous* clock (CKMODE) so the converter ran off the 280 MHz AHB bus / 4 = 70 MHz, out of range, ignoring the PLL2P (~98 MHz) kernel clock `SystemClock_Config` already provisions as `ADCSEL`; and (2) it never set the channel pre-selection (`PCSEL`), so input 4 (PC4, the battery sense pin) was disconnected and conversions returned ~0. Fixed by switching to the asynchronous PLL2P clock (CKMODE/PRESC = 0, matching the reference's `ADC_CLOCK_ASYNC_DIV1`), setting `BOOST = 0b11` (mandatory above 25 MHz, as the HAL's `ADC_ConfigureBoostMode` computes for this clock), and adding `PCSEL` for channel 4. That restored a reading but it sat ~25% low and drifted: (3) a bogus `ADCAL` software calibration was subtracting a ~930-count offset from every conversion — removed (the reference/Retro-Go run uncalibrated and read correctly); and (4) the 810.5-cycle sampling window straddled active LTDC/OSPI and integrated supply ripple into noisy reads that the discharging min-latch then pinned to their lowest excursion — shortened to 1.5 cycles (matching the reference, low-impedance sense node) and `board_get_battery_raw` now averages 16 conversions. Confirmed apples-to-apples by reading the same ADC `DR` over SWD while Retro-Go ran (steady ~13350): our reading went 0 → 13248–13311 and the header now shows a stable 100%. New diagnostics `scripts/debug/read_battery.py` (ADC state + live/halted sampling) and `scripts/debug/launch_retrogo.py` (CORE-magic re-launch for cross-firmware comparison).
- **Partition Viewer: reliable Retro-Go detection + version banner, correct Bank 1 sizes (hardware-verified).** Retro-Go is identified by scanning from its fixed base (`RETROGO_BASE`) for the version banner literal (`"Retro-Go SD v"`), replacing the fragile fixed-offset `0x400`/`01 00 00 00` static probe (a generic signature off the 16 KiB scan grid that false-matched and went stale). The new `partition_retrogo_version()` returns the raw on-flash version token, shown under Details (e.g. `v1.3.1-84-g905d6615`), cached per selection. The Chainloader partition size was a stale `0x8000` (32 KiB) literal and now derives from `RETROGO_BASE - BANK1_BASE` (40 KiB), and Retro-Go fills from its base to the end of Bank 1 (it is always the last occupant) instead of a 16 KiB grow-scan that left an 8 KiB tail gap.
