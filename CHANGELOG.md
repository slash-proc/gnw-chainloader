# Changelog

A terse, bullet-point record of completed bug fixes and tasks, newest first. Keep each entry to one line and link the commit(s) that landed it. (Entries below predate this convention, so they are not all commit-linked.)

- **Module source tree renamed `drivers` -> `modules` to match the device paths.** The on-device LittleFS paths became `/modules` (FS-driver subfolder `/modules/filesystems`) with the installer work, so the source tree + build now match: `src/drivers` -> `src/modules`, `src/drivers/fs` -> `src/modules/filesystems`, each module's `driver_entry.c` -> `module_entry.c`, the PIE entry symbol `init_driver` -> `init_module` (in `module.ld` + every module), and the Makefile `DRIVER_*` vars -> `MODULE_*` (`build/drivers` -> `build/modules`). `vfs_driver_t` / `theme_driver_t` keep their names (vtable descriptors a module fills, the classic driver sense). Core unchanged at 38312 B; clean build. (`b7fdafe`)
- **SD install becomes a transient installer module that handles both languages AND modules (ABI Stage 2).** The SD->LittleFS install moved out of the resident language module into `/modules/installer.bin` (`MOD_FLAG_TRANSIENT`): the core loads it on demand, scan -> confirm -> commit -> reclaims the slot. One generic descriptor-driven path (`artifact_kind_t`) installs both language packs and PIE modules, gated by the running core's ABI (the install gate mirrors the loader's load gate). The core shows a per-class confirm ("Install N language(s) from SD?" / "Install N module(s) from SD?"); languages apply live (i18n re-discovery), installed modules apply on the next boot. The copy streams through one shared in-core `vfs_copy_open_file` (4 KiB buffer) that the file browser also uses, so even large module images install without a whole-file buffer (the installer's RAM footprint dropped ~17 KB -> ~1.7 KB). The language module sheds its install code + 16 KB buffer and exposes `rediscover()`. The FS-driver deps (FAT for SD reads, lfs_rw for LittleFS writes) are loaded resident before the installer's pool mark, so reclaiming the transient slot can't free a driver the vfs still points at (a use-after-free that was the real commit bug). The on-device module folder was renamed `/drivers` -> `/modules` (and `/modules/fs` -> `/modules/filesystems`) since modules outgrew the original FS-driver system. Hardware-verified: `installer_test --kind all` 6/6 (language + module each offered with class-aware wording and reinstalled from the SD), `theme_lang_test` + `retrogo_return_test` green. - **Fast framebuffer capture reworked into a D2-resident tiled hardware-JPEG codec.** The codec (`src/fastcap/`) now lives entirely in the unused D2 AHB-SRAM1 bank (`0x30000000`) — isolated from AXI by construction, so it can never collide with the app, framebuffers, or the PIE module pool (an earlier AXI-resident layout overwrote a loaded module and bus-faulted). It tiles the screen into 32×16 tiles, detects change with a per-tile FNV-1a hash (no prev-frame buffer — the 600 B hash table is the whole change-detection state), and hardware-JPEG-encodes only changed tiles. Header-strip (only the keyframe's first tile carries the ~600 B JPEG header; the host caches it and re-wraps later headerless scans), silent-skip (unchanged frames aren't sent — a `skipped` count drives host replay — with a `tile_count=0` timing anchor every 30 silent frames), and a **runtime JPEG-quality cell** (`FASTCAP_QUALITY` 1..100, default 98 — no rebuild to change it). async (low-latency, drops frames) / sync (frame-perfect) modes via `FASTCAP_MODE`. Unified host tool `scripts/debug/fastcap.py` (capture/grab/analyze) on the shared `common/device.py` SWD backend with gamepad-first input, a live OpenCV preview + FPS overlay, and mp4 assembly; payload cap grown to 47 KB by relocating the code region. Capture is throughput-bound (~50–70 KB/s effective SWD, *not* clock-bound), so distinct fps ≈ throughput ÷ bytes-per-frame and quality is the live lever (q80 ~30 fps soft, q100 ~5 fps crisp; sync mode for smooth high-quality clips). Subsumes and deletes `capture.py` and `fastcap_frames.py`. See [docs/fast-framebuffer-capture.md](docs/fast-framebuffer-capture.md).
