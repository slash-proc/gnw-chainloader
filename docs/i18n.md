# Internationalization (i18n)

How the chainloader UI is translated and rendered in non-English / non-Latin
scripts, and how to add or change a language. This is the engineering reference;
the plain-language summary is the "Multi-Language Menus" feature in
[README.md](../README.md).

> **Architecture update (lean core):** the i18n implementation — discovery,
> `.lang`/`.fnt` loading, UTF-8 rendering (`font_ext`), and switching — now lives in
> the **PIE language module** (`src/modules/language/` → `/modules/language.bin`),
> not the firmware. The core bakes only the ASCII font + English string table and a
> thin `ui/i18n.c` shim that loads the module when present (at boot, whenever the
> file exists) and delegates; with the module absent, the core renders English-ASCII
> only and the menu still works. The module fills the core's `gui_set_ext_glyph`
> render hook and `strings_set_active` table; it no longer performs the SD install
> (that moved to a transient installer module, see below) and instead exposes a
> `rediscover()` the core calls after an install lands. The host↔module contract is
> `src/chainloader/system/language.h`. The pack/font formats and the "add a language"
> workflow below are unchanged; only their *home* moved.

## Goal & invariants

Make every user-facing menu string translatable — including scripts the in-core
font can't draw (accented Latin, CJK) — while staying a lean emergency-tool
chainloader. **Only English is baked into the firmware.** Every other language is
**data** discovered on the device filesystem, so adding a language never touches
the binary. **Stability is law:** the menu must stay reachable in every degraded
state (in-core English + ASCII is always present), and a missing/corrupt asset can
never block the boot path.

## Architecture (three layers, each degrades to the one below)

1. **In-core, always present.** A baked **Fusion Pixel 12px monospaced** ASCII
   font (`ui/gui_font.c`, `0x20`–`0x7E`, ~1 KB) and the English string table
   (`ui/strings.c`). Emergency English in the device's own flash.
2. **Shared script fonts (on the filesystem).** Per-script `.fnt` blobs supply the
   non-ASCII glyphs (`ui/font_ext.c` reads one whole into RAM, binary-searched).
   The **`latin` base** is loaded once at startup and left resident; per-language
   non-Latin scripts (CJK) occupy a second slot. Latin-script languages reuse the
   resident base — no per-switch reload.
3. **Language packs (on the filesystem).** Per-language `.lang` files hold the
   translated strings **and self-describe** (each carries its own locale code +
   endonym + script). `ui/i18n.c` discovers them and loads the active one.

`gui_glyph(cp)` resolves: in-core ASCII → active script font → shared Latin base →
`'?'`. `tr(id)` resolves: active pack → in-core English (per-string fallback).

## Runtime language list (discovery, not a registry)

At boot, once the filesystems are up, `i18n_init()` builds the language list:
**English (index 0, in-core)** plus every valid pack found by scanning
`/i18n/<code>.lang` on LittleFS (`vfs_lfs_enum_langs` reads each 76-byte header for
its code/endonym/script). The list is **sorted by locale code** with English
pinned first. There is no compiled-in registry — a language never built into the
firmware appears automatically once its pack is present, as long as its ABI
matches. **ABI gating is core-authoritative:** `vfs_read_lang_lfs` /
`vfs_lfs_lang_version` / `vfs_lfs_enum_langs` check each pack header against the
core's own `STRINGS_ABI_VERSION` (shared `src/common/abi.h` `pack_abi_ok`), not a
value the language module passes in, so a stale module plus stale packs can never
agree and load garbage; a mismatched pack is skipped and falls back to English.
The active language is persisted **by code** in `/i18n/.active`
(`i18n_persist_active`), so the list can grow or shrink without disturbing the
setting; an absent/unmatched code falls back to English.

## Rendering

- **Font cell.** Glyphs are a `width`×`h` 1bpp bitmap, MSB-first. External blobs
  carry their own `ref_top`; the renderer draws each at `y + (ref_top - GUI_FONT_REF_TOP)`,
  so any blob shares the in-core baseline.
- **UTF-8 + proportional.** `gui_utf8_next()` decodes 1–4 byte scalars; `gui_text_width()`
  sums per-glyph advances. These helpers live in `gui_text.c` — **HAL-free**, so
  they run on the host (`make test_host`).

## File formats (little-endian)

Cooked into `build/i18n/` by the scripts below; `build/i18n/` is a **1:1 mirror of
the SD card's `/i18n/`** (`<code>.lang` + `fonts/<script>.fnt`).

### `<script>.fnt` — `scripts/build/cook_font.py blob`
```
u32 magic 'FNT1'; u16 glyph_count; u8 cell_h; u8 ref_top; u32 bitmaps_off
u32 codepoints[]   // sorted ascending — binary searched
u8  widths[]       // padded to a 4-byte boundary
u32 offsets[]      // byte offset of each glyph's bitmap
u8  bitmaps[]      // each glyph: ceil(w/8)*cell_h MSB-first bytes
```
The shared **`latin`** blob covers Latin-1 Supplement + Extended-A/B + Greek +
Cyrillic; CJK blobs are **subset** to exactly the codepoints used by that script's
translations **plus the language endonyms**.

### `<code>.lang` — `scripts/build/cook_lang.py build` (self-describing, magic `'LNG2'`)
```
u32 magic 'LNG2'; u16 abi; u16 str_count; u32 version
char script[16]; char code[16]; char endonym[32]      // 76-byte header
u32 offsets[str_count]   // blob offset, or 0xFFFFFFFF = English fallback
u8  blob[]               // NUL-terminated UTF-8 strings
```
`lang_parse()` rejects an `abi`/`str_count` mismatch (the core↔pack wire contract
is `STRINGS_ABI_VERSION` + the `string_id_t` order in `ui/strings.h`). The magic
`'LNG2'` cleanly invalidates the previous format. `version` is the per-language
**content** version (from `langs.json` `"version"`, default 1): the SD install
installs a pack only when its version is newer than the LittleFS copy.

## SD → LittleFS install (transient installer module)

The canonical pack lives at `/i18n/<code>.lang` on **LittleFS**; runtime reads are
**LittleFS-only** (`vfs_read_lang_lfs`) so switching language never scans the slow
SD card. Installing from the SD is no longer done by the language module. It now
lives in a **transient PIE installer module** (`/modules/installer.bin`, flagged
`MOD_FLAG_TRANSIENT`): the core loads it on demand, runs it, then reclaims its pool
slot (`mod_pool_mark` / `mod_pool_reset`).

The installer drives **two artifact classes through one generic,
descriptor-driven path**: language packs (`/i18n/<code>.lang` plus their
`/i18n/fonts/<script>.fnt`) and PIE modules (the fixed `/modules/*.bin` list). For
each candidate it applies an **install gate that mirrors the loader's LOAD gate**:
magic plus the running core's ABI for that class, read from the core's own ABI
values, so a stale artifact can never slip in.

1. If an SD card with an `/i18n` (or `/modules`) folder is present (cheap probe — no
   eager module load otherwise), load the **LFN FAT-RW module**
   (`/modules/filesystems/fatfs.bin`) so the 4-char `.lang`/`.bin` extensions are
   readable (the in-core FAT is 8.3-only).
2. Enumerate the SD candidates; for each, peek its header (header-only reads check
   the version) and compare to the LittleFS copy. If newer/absent (and the gate
   passes), it is a commit candidate.
3. The core shows a **per-class confirm pop-up at boot**: "Install N language(s)
   from SD?" or "Install N module(s) from SD?" (combined when both apply). On accept
   it commits, writing **only the `/i18n` and `/modules` data folders** (via
   `vfs_main_lfs`), never a flash bank or the boot path, bounded and fail-safe
   (any error leaves English active and the menu reachable). LANGUAGES apply live
   (the core calls the language module's `rediscover()` to re-discover and re-apply
   i18n); installed MODULES apply on the next boot.

The copy **streams** through the shared in-core `vfs_copy_open_file` (a single 4 KiB
buffer, no whole-file buffer), the same copy loop the file browser uses; modules
reach it via `vfs_copy_sd_to_lfs`. So a user copies `build/i18n/` onto the SD's
`/i18n/` (no renaming), confirms the prompt, and the device installs it.

## Settings & persistence

The active language is the locale **code** in `/i18n/.active` (a few bytes on
LittleFS), written on Settings exit (debounced, so rapid cycling doesn't write a
file per tick) and read at boot to restore the choice. The **Settings → Language**
value-selector (`menu.c`, `ui_lang_cycle`) switches live (loads pack + font,
refreshes cached titles). English (index 0) is the default if `/i18n/.active` is
absent or names a language with no pack.

## Build & deploy

```
make i18n        # cook packs (build/i18n/<code>.lang) + fonts (build/i18n/fonts/<script>.fnt)
make push_i18n   # dev shortcut: gnwmanager push all assets to the device /i18n/
make test_host   # host unit tests for the text layer + the .fnt reader
```
`make i18n` is **not** part of `all` — the firmware only needs the committed
in-core `ui/gui_font.c` + `ui/strings.c`. The cooked `.fnt`/`.lang` files are build
artifacts (`build/i18n/`, a clean SD mirror); `.chars` font-subset intermediates
go to `build/i18n_tmp/` to keep the mirror clean. For the **real user flow**, copy
`build/i18n/` onto an SD card's `/i18n/` and let the device install it; `push_i18n`
is just the dev shortcut that writes straight to LittleFS.

## Sources (`i18n/`)

- `i18n/fonts/*.otf` — Fusion Pixel, per script (OFL-1.1; keep `LICENSE/` + `OFL.txt`).
- `i18n/lang/langs.json` — the **build manifest**: `{code, english, endonym, script}`
  per language (locale codes: `de_DE`, `en_US`, `zh_CN`, `zh_TW`, …). The device no
  longer has a compiled-in registry; this only drives `make i18n`.
- `i18n/lang/<code>/strings.json` — `{STR_id: "translation"}`; omitted ids fall back
  to English.

## Adding a language

1. Add an entry to `langs.json` (pick the `script`: an existing
   `latin`/`ja`/`ko`/`zh_hans`/`zh_hant`, or add a new Fusion OTF).
2. Create `i18n/lang/<code>/strings.json` (copy `_reference.json`, translate). Keep
   placeholder shapes (trailing `": "`/space on prefixes, button letters, literal
   `RETRO-GO` / `OFW` / `KB`).
3. `make i18n`, then copy `build/i18n/` to the SD (or `make push_i18n`). **No
   firmware change** — the device discovers it. A brand-new script needs its
   `.fnt` cooked + present, nothing more.

## Adding / changing a string

Add the `STR_*` id to `string_id_t` **and** the English table (`ui/strings.{h,c}`)
— append at the end and route it through `tr()`. Bump `STRINGS_ABI_VERSION` only on
a reorder/removal; re-cook all packs.

## Verification status

Hardware-verified: English-only baked + filesystem discovery (English + `de_DE` +
`zh_CN`), sort-by-code, `LNG2` read with Latin (`Deutsch`, `ü`) and CJK (`中文`)
render, persist-by-code across a reboot, and the **Retro-Go quit → launcher
round trip** (the magic is consumed and the menu is reached in the persisted
language). Host tests (`make test_host`) cover UTF-8 decoding, glyph resolution +
proportional width, the English/active-pack fallback, and the `.fnt` reader.

## Out of scope / follow-ups

On-device OTF rasterization (pre-extracted instead); RTL/bidi/shaping. The
SD-install scan runs on every boot when an `/i18n` (or `/modules`) folder is present
(bounded; only prompts when something is newer) — it could be skipped on a warm
Retro-Go return for extra snappiness, a possible future optimization.
